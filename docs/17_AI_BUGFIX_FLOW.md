# 17. AI Bugfix Flow v2 — інструкція yurii

> **Кому це:** yurii — єдина людина, що просуває цей конвеєр (лейбли, approve, merge).
>
> **Що це:** як завести дрібний баг/покращення так, щоб він пройшов конвеєр тріаж → фікс → рев'ю без ручної Claude Code-сесії, і що робити на кожному лейблі.
>
> **Архітектурне рішення:** ADR-035 (`docs/03_DECISIONS.md`). Джерело промпту: `docs/promts/master-ai-bugfix-flow.md`.

---

## 1. Життєвий цикл одного issue

```
[issue за формою] → лейбл ai-fix → (ai-triage.yml) → ai-plan-ready
                                                     → ai-architectural (STOP, звичайний трек)
                                                     → ai-need-info (відповісти → знову ai-fix)

ai-plan-ready → (прочитав план) → лейбл ai-approved → (ai-fix.yml) → draft PR
                                                                    → ai-architectural (STOP, PR немає)

draft PR → docs/promts/ai-review-local.md (одна команда, локально) → merge (ти)
```

Кожна стрілка — ручна дія з твого боку, окрім двох `(ai-*.yml)` — це GitHub Actions,
запускаються самі, щойно з'являється відповідний лейбл.

## 2. Як завести issue

Дві форми (`.github/ISSUE_TEMPLATE/`): **🐛 Баг** (`bug_report.yml`) і
**✨ Покращення** (`improvement.yml`). Заповни поля — вони структуровані
навмисно (Де / Кроки / Очікував / Побачив / Скріншот / Пристрій), тріаж це
читає першим.

**Лейбл `ai-fix` НЕ ставиться формою автоматично** (свідомо, ADR-035 §5.1) —
постав його сам, ПІСЛЯ того, як прочитав issue і вирішив, що варто спробувати
автоматичний фікс.

## 3. Тріаж (`ai-triage.yml`) — що читати

Приходить коментар з одним із трьох результатів:

- **`ai-plan-ready`** — root-cause гіпотеза + план (файли у скоупі, який
  регресійний тест додасться, рекомендована модель) +, за потреби, розділ
  «Опитування» (варіанти з ★ рекомендацією). **Прочитай план ПОВНІСТЮ ПЕРЕД**
  тим, як ставити `ai-approved` — це і є твоя точка контролю, аналог
  «прочитай ADR перед B-промптом» з `docs/15` §0 п.3.
  - Погоджуєшся з планом (і, якщо було «Опитування» — обрав варіант,
    зазвичай ★) → постав лейбл `ai-approved`.
  - Не погоджуєшся → коментар з правками, лейбл лишається `ai-plan-ready`
    (жоден workflow на нього не реагує — можеш перезапустити тріаж, знявши
    і знову поставивши `ai-fix`, або просто продовжити вручну).
- **`ai-architectural`** — тріаж вирішив, що це не дрібний фікс: зачіпає
  §7/§13-інваріанти, спільні шари, схему БД, infra, або Registry-зону
  (§6 нижче). Коментар пояснює чому + посилання на трек `docs/02_ROADMAP.md`.
  Це СТОП для цього конвеєра — переведи задачу у звичайний трек (додай
  пункт у чергу відповідного треку, або створи master-run промпт для
  more складного випадку).
- **`ai-need-info`** — 1-3 конкретні питання. Відповідай прямо в issue,
  потім знову постав `ai-fix` (новий прогін тріажу побачить твою відповідь
  через `gh issue view --json comments`).

## 4. Фікс (`ai-fix.yml`) — що читати

Після `ai-approved` — за кілька хвилин або draft PR, або коментар
`ai-architectural` (фікс на півдорозі зрозумів, що план вимагає забороненого
шляху — заборонені шляхи `packages/db/src/migrations/**`, `infra/**`,
`.github/**`, `CLAUDE.md`, `docs/03_DECISIONS.md`,
`docs/12_TEMPLATE_CONTRACT.md`, `packages/db/src/schema.ts`,
`workers/cad/tests/snapshots/**` — ADR-035 §5.4). У другому випадку PR-у
немає — це очікувано, не помилка workflow'у.

Якщо є draft PR — читай його опис (план + «Як перевірити очима» +
«Опитування», якщо лишились варіанти) і переходь до §5.

## 5. Локальне рев'ю — одна команда

У СВІЖІЙ Claude Code-сесії (не та, що читала issue раніше — не обов'язково,
але рекомендовано для чистоти контексту):

```
Виконай docs/promts/ai-review-local.md для PR #<N>
```

Це запускає `agy`/Gemini як незалежного Рев'юера і Тест-інженера
(read-only), Claude верифікує й оформлює прийняті тест-кейси. Результат —
коментар на PR з вердиктом `agy` (дослівно, позначка «Gemini Reviewer
(unedited)») + підсумок від Claude. **Merge — після того, як прочитав ОБИДВА
вердикти** (Claude-Будівельник написав код; Gemini — незалежна перевірка;
розбіжність між ними — цінний сигнал, не привід ігнорувати один з них).

## 6. Конфлікт з Run 7 (Template Registry migration)

Поки триває Registry-міграція (трек T2, `docs/02_ROADMAP.md`,
`docs/promts/master-registry-track.md`) — файли студій/шаблонів
(`apps/web/src/components/*-{studio,editor,viewport}.tsx`,
`workers/cad/flatcraft_cad/templates/*.py`, `packages/types/src/templates/*.ts`)
активно переписуються по одному шаблону за раз (ADR-033). Будь-який ai-fix
у цій зоні ризикує зустрічними правками з Registry-PR того ж файлу.

**Правило:** поки T2 активний, тріаж ЗОБОВ'ЯЗАНИЙ класифікувати фікси у
перелічених вище файлах як `ARCHITECTURAL`, незалежно від того, наскільки
дрібним виглядає баг. Коментар тріажу має явно посилатись на це правило
(«Registry-міграція в процесі, `docs/17` §6») — так, дрібний typo-фікс теж
іде у звичайну чергу, а не через ai-fix.

Коли T2 завершиться (Registry — ЄДИНЕ джерело шаблонів) — це правило
відпадає, приберемо цей розділ окремим PR.

## 7. Правило «дані → issue»: digest і Sentry

`docs/14` §4.4 фіксує механізм самовдосконалення: **кожен пункт щотижневого
digest'а (Discord) і кожен новий Sentry-issue стає GitHub-issue АБО явно
позначається «accepted noise»** у digest-треді. Це джерело кандидатів для
`ai-fix` — не єдине (руками заведені баги теж ідуть тим самим шляхом), але
основне постачання. Заводь issue за формою (§2), навіть якщо джерело —
автоматичний digest-пункт: людський опис («Де / Кроки / Очікував / Побачив»)
важливий для тріажу, сирий Sentry-stacktrace чи цифра з digest'а — недостатньо.

## 8. Одноразове налаштування (перед першим використанням)

1. `claude setup-token` (локально) → скопіювати токен.
2. GitHub → Settings → Secrets and variables → Actions → New secret:
   `CLAUDE_CODE_OAUTH_TOKEN`.
3. Створити лейбли (готові команди — у description setup-PR).
4. **Acceptance-прогін**: взяти РЕАЛЬНИЙ дрібний баг зі свого списку, пройти
   повний цикл §1-§5 до merge, перевірити фікс очима на staging.

## 9. Типові фейли

| Симптом                                               | Причина                                                                                | Дія                                                                                |
| ----------------------------------------------------- | -------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------- |
| Тріаж не запустився після `ai-fix`                    | Лейбл поставив користувач без write-доступу — GitHub це фізично не дозволяє (§1 gate)  | Не проблема цього флоу; постав лейбл сам                                           |
| `ai-fix.yml` впав на «Verify forbidden paths»         | Фікс зачепив заборонений шлях (permissions.deny не спрацював, backstop-скрипт впіймав) | Прочитай лог job'у, видали/закрий PR і гілку вручну, заведи issue у звичайний трек |
| Draft PR є, але тестів немає                          | Bug у промпті fix-стадії (регресійний тест мав бути обов'язковим)                      | Коментар на PR + `ai-review-local.md` все одно зловить це на кроці 2 (agy)         |
| `agy` переписав чужий файл поза `docs/promts/inputs/` | Відомий клас інциденту (2× Master Run 8) — `write_file(*)` ширший за задум             | `git restore`, перевір byte-identical, продовж; рядок у `agy-stats.md`             |
