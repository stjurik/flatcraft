# 14. Architecture Evolution — від sheet-metal MVP до мультипроцесної платформи

> **Статус:** Proposal (2026-07-05). Аналіз кодової бази + рекомендації. Жодне рішення тут не є прийнятим, поки не зафіксоване як ADR у `docs/03_DECISIONS.md`.
> **Мета документа:** відповісти на три стратегічні запити yurii:
>
> 1. кожен новий виріб має перевикористовувати попередні напрацювання (зараз 2 категорії відображаються/обраховуються по-різному — що буде при 20?);
> 2. платформа має розширюватись на нові виробничі процеси (3D-друк, мехобробка) без переписування;
> 3. платформа має збирати зворотний зв'язок під час роботи (логи, метрики, виробничий фідбек) і на його основі самовдосконалюватись.

---

## 1. Діагноз поточного стану (факти з кодової бази, 2026-07-05)

### 1.1. Що вже добре і що зберігаємо як фундамент

- **Single source of truth:** Zod-схеми у `packages/types`, bend-матриця в одному YAML (`packages/cad-engine/data/bend-machine-esi.yaml`), який читають і TS, і Python.
- **Детермінізм CAD** (байт-у-байт DXF/PDF) + снапшот-тести — це рідкісна перевага; вона робить будь-який рефакторинг безпечним.
- **Трирівнева валідація** (клієнт UX → Fastify gate → Python parity, ADR-019/026) — правильний паттерн, його треба масштабувати, а не міняти.
- **Products-шар** (ADR-027..031): preset поверх базового шаблону — вже перший крок до «нового виробу без коду» (3-й виріб = seed entry + PNG).
- **Культура «інцидент → тест»** (Hotfix 2.10.e → property-тести ×1000) — це і є зародок self-improvement loop, лишилось дати їй паливо (дані з продакшену).

### 1.2. Проблема A — «вартість нового шаблону» надто висока

Додавання шаблону №7 сьогодні торкається **~12 місць у 5 workspace**:

| Workspace        | Файли на один шаблон                                                                                                                                            |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `packages/types` | `templates/<slug>.ts` (Zod) + запис у discriminatedUnion `ExportRequest`                                                                                        |
| `packages/ui`    | shape-builder у `3d-viewport/geometry.ts` + `<Slug>Scene`                                                                                                       |
| `apps/web`       | `<slug>-studio.tsx` + `<slug>-editor.tsx` + `<slug>-viewport.tsx` + гілка у `template-thumb.tsx` + hardcoded union `TemplateStudioSlug` у `template-studio.tsx` |
| `workers/cad`    | `templates/<slug>.py` + гілки в unfold/dxf/pdf/validate + снапшоти                                                                                              |
| `packages/db`    | seed entry                                                                                                                                                      |

Наслідок: кожен шаблон — частково копі-паст, і саме тому «різні вироби по-різному відображаються і, можливо, обраховуються». Дрейф між копіями — це не баг конкретного шаблону, це **системна властивість поточної структури**. При 20 шаблонах ×3 процеси вона стане некерованою.

`TemplateStudio` (Phase 3.0 PR 4) уже узагальнив каркас студії через render-props — але editor/viewport/scene досі пишуться вручну на кожен шаблон.

### 1.3. Проблема B — sheet-metal зашитий у словник платформи

`unfold`, `bends`, `bend_radius`, DXF-шари, bend-матриця — присутні на всіх рівнях (types, api, worker, ui). Немає поняття «виробничий процес» як сутності. Додавання 3D-друку сьогодні означало б паралельну копію всієї вертикалі з `if process == ...` у кожному шарі.

### 1.4. Проблема C — зворотного зв'язку фактично немає

| Канал                                                             | Стан                                                                                                                                                                                      |
| ----------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| pino-логи (api)                                                   | ✅ є, з PII-redact — але живуть у stdout контейнера, ніхто їх не агрегує і не читає                                                                                                       |
| Sentry (web/api/worker)                                           | ✅ Phase 3.3 PR 3 (#56, 2026-07-06) — SDK + `beforeSend` PII у трьох сервісах; **активовано на staging 2026-07-12** (2 проєкти: `flatcraft-web` + `flatcraft-server`, PR #63 vault + #65) |
| Продуктова аналітика (Plausible/Umami)                            | ✅ Phase 3.3 PR 5 (#58, 2026-07-06) — Umami self-hosted (ADR-032 §4), cookie-less, воронка `catalog→…→export_done`; **активовано на staging 2026-07-12** (PR #63)                         |
| Історія експортів                                                 | ✅ Phase 3.3 PR 2 (#55, 2026-07-06) — `exports` персистується у Postgres (drizzle-репо замість in-memory JobStore); `events` (append-only) з Zod-payload'ами і `session_hash`             |
| Виробничий фідбек («деталь зроблена/брак/відхилення мм»)          | ❌ R-01 mitigation 4 («report deviation API») — Phase 3.4, у роботі (master-registry-run WP3)                                                                                             |
| Зв'язок «валідаційна помилка → рішення про параметризацію» (R-10) | ✅ Phase 3.3 PR 5 (#58) — воронка Umami + подія `validation_error_shown` (constraint = code першого issue) → щотижневий digest (#60)                                                      |

Платформа **більше не сліпа** (стан 2026-07-12): три з шести каналів закриті у Phase 3.3, ще один (виробничий фідбек, R-01) — у роботі. Єдиний невідомий канал (`pino → агрегація`) свідомо залишено на пізніше: логи стриміняться у `docker compose logs`; агрегатор (Loki/OTel) — тільки якщо цього стане мало (14 §5 антицілі).

## 2. Рекомендація 1 — Template Registry: єдиний контракт шаблону

**Ціль:** новий шаблон = **1 TS-модуль + 1 Python-модуль + снапшоти**. Нуль змін у `apps/web`, `apps/api`, нуль нових React-компонентів.

Суть — задекларувати контракт `TemplateDefinition` і зібрати всі шаблони в один реєстр:

```ts
// packages/templates (новий пакет) або розширення packages/types
interface TemplateDefinition<T> {
  slug: string;
  process: "sheet_metal"; // див. Рекомендацію 2
  schema: z.ZodType<T>; // єдина Zod-схема (вже є)
  defaults: T;
  ui: {
    sceneBuilder: (params: T, thicknessMm: number) => ShapeCommands; // з geometry.ts
    thumb?: ThumbDescriptor; // замість гілок у template-thumb.tsx
  };
  validators: ProfileValidator[]; // validateProfile-паритет (ADR-026)
}
export const TEMPLATE_REGISTRY: Record<string, TemplateDefinition<unknown>>;
```

Що це дає:

1. **Сторінка студії стає повністю data-driven:** `/templates/[slug]` бере definition з реєстру → AutoForm рендерить editor зі схеми (це вже вміє `parameter-form`), generic viewport бере sceneBuilder. 18 per-template файлів у `apps/web` зникають.
2. **Python-дзеркало** вже майже існує (`templates/base.py`, конвенція params/build/unfold) — лишається формалізувати реєстр і паритет-тест «кожен slug з TS-реєстру має Python-модуль і навпаки».
3. **Conformance-suite** — параметризований тестовий набір, який ганяє **кожен** зареєстрований шаблон через однакові перевірки: schema parity TS↔Python (property-based, як у 2.10.e), детермінізм DXF/PDF, render-gate на невалідних параметрах, e2e smoke (відкрити студію → змінити параметр → export). Шаблон фізично не може бути «доданий наполовину» — реєстр без conformance = червоний CI.
4. **Уніфікація відображення:** «по-різному відображаються» лікується не полюванням на окремі баги, а тим, що спільний код-шлях стає єдиним можливим.

Ризики міграції: низькі — геометрія і схеми не змінюються, лише перекладання у реєстр; e2e-набір (92+ тестів станом на 2.9.f) — це страхувальна сітка. Мігрувати по одному шаблону на PR.

## 3. Рекомендація 2 — шар «Виробничий процес» (Process layer)

**Ціль:** 3D-друк і мехобробка додаються як **плагіни-вертикалі**, а не форки кодової бази.

Нова ієрархія сутностей платформи:

```
Process (sheet_metal | fdm_print | cnc_machining)
  └─ Template (параметрична модель у межах процесу)
       └─ Product (preset шаблону, ADR-027)
            └─ Draft → Export → Artifact[] → Feedback
```

Процес = пакет з чотирьох речей:

| Компонент              | sheet_metal (є)             | fdm_print (майбутнє)                      | cnc (майбутнє)                             |
| ---------------------- | --------------------------- | ----------------------------------------- | ------------------------------------------ |
| Capability-spec (YAML) | `bend-machine-esi.yaml`     | build volume, матеріали, min wall, nozzle | робоча зона, інструменти, min radius фрези |
| Пак валідаторів        | bend matrix, profile, sheet | overhang/wall/bridging                    | tool access, fillet ≥ R фрези              |
| Пак експортерів        | DXF + PDF (+STEP)           | STL/3MF + PDF-паспорт                     | STEP + PDF-креслення                       |
| Worker-модуль          | CadQuery unfold/export      | slicer-agnostic mesh export               | CAM-ready STEP                             |

**Що зробити ЗАРАЗ (дешево, без жодного нового процесу):**

1. **Узагальнити словник там, де він протікає назовні:** `artifacts` як список `{kind, mime, r2_key}` замість фіксованих ключів `{dxf, pdf, step}`; `templates.process_slug` (поки константа `sheet_metal`); `ExportJob.process` у payload черги.
2. **Зафіксувати ADR** з цією ієрархією і правилом: жодна нова фіча не додає sheet-metal-специфіку у спільні шари (types/api/web) — тільки у процес-пакет.
3. **Нічого не будувати наперед** (YAGNI): ні абстрактних slicer-інтерфейсів, ні порожніх cnc-модулів. Перший реальний новий процес — FDM-друк, бо він найдешевший (немає розгортки, немає bend-матриці, STL-експорт з CadQuery тривіальний) і найкраще лягає на DIY-аудиторію.

## 4. Рекомендація 3 — Observability і цикл самовдосконалення

**Принцип: кожен експорт — це експеримент, результат якого платформа зобов'язана зібрати.** Це найважливіша з трьох рекомендацій, і саме її варто робити **першою**: без даних рішення про нові шаблони/процеси прийматимуться наосліп.

### 4.1. Рівень 1 — технічна телеметрія (бачити, що ламається)

- **Sentry у web + api + worker** (Phase 5.1) — підняти пріоритет до «перед публічним soft-launch». `beforeSend` PII-фільтр — уже інваріант CLAUDE.md §8. Це єдиний спосіб дізнатись про краші R3F на реальних телефонах (R-02 acceptance зараз неперевірюваний).
- **Таблиця `events`** (append-only, без PII): `id, ts, event_type, template_slug, process, params jsonb, error_code, duration_ms, session_hash`. Пишуть її Fastify-hooks (export_requested / validation_rejected / export_completed / export_failed) і worker (cad_started / cad_completed з тривалістю). Параметри виробу — не PII, це найцінніший датасет платформи.
- **JobStore → Postgres:** таблиця `exports` вже описана у `docs/05_DATA_MODEL.md` — реалізувати. Історія експортів = сировина для калібрування, статистики і аудиту (R-12 mitigation 5 — аудит R2 — стане можливим).
- **Бюджети CLAUDE.md §9 стають вимірюваними:** p95 export з `events.duration_ms`; web-vitals (FCP/TTI/mesh-update) — custom events у аналітику.

### 4.2. Рівень 2 — продуктова аналітика (бачити, де користувач страждає)

- **Umami self-hosted** (ADR-032 §4 — фінальне рішення 2026-07-05; було «Plausible або Umami»), cookie-less, GDPR-friendly.
- Воронка: `catalog → studio_opened → param_changed → validation_error_shown → export_clicked → export_done`.
- **Ключова метрика — `validation_error_shown` з розбивкою по constraint:** який ліміт найчастіше блокує користувачів → який шаблон потребує ширшої параметризації. Це прямий вхід для R-10 (шаблони не задовольняють реальні потреби) — зараз він оцінюється інтуїцією.

### 4.3. Рівень 3 — виробничий фідбек (бачити, чи зійшлось у металі)

Найцінніший і найунікальніший рівень — жоден лог його не замінить:

- **QR на PDF вже є** (Phase 2.9). Спрямувати його на `/f/{export_id}` — мобільна форма з трьох полів: «виріб виготовлено: так/ні», «розміри зійшлись: так / відхилення (мм, де саме)», «коментар». Без реєстрації, 30 секунд у цеху.
- Таблиця `export_feedback (export_id FK, outcome, deviation_mm, bend_no, comment, created_at)` — зв'язана з повним знімком параметрів через `exports`.
- Ці дані замикають **R-01 (K-фактор)**: `tools/scripts/calibrate-k-factor.py` вже існує — тепер у нього з'явиться вхід. Достатньо 10–20 репортів по матеріалу, щоб калібрувати таблицю краще за будь-який пілот.

### 4.4. Замикання циклу (self-improvement loop)

```
   логи/події/фідбек                    аналіз                     дія
┌──────────────────────┐   ┌───────────────────────────┐   ┌──────────────────────┐
│ Sentry issues        │   │ Щотижневий digest (cron   │   │ пункт у Roadmap /    │
│ events (validation,  │ → │ у worker → Discord        │ → │ hotfix + обов'язковий│
│  durations, failures)│   │ webhook): top-5 помилок,  │   │ регресійний тест     │
│ export_feedback      │   │ повільні експорти,        │   │ (культура 2.10.e)    │
│ web-vitals           │   │ deviation-репорти         │   │ калібрування YAML    │
└──────────────────────┘   └───────────────────────────┘   └──────────────────────┘
```

- Digest — це «звіт платформи самій собі»: дешевий (один cron + webhook, інфраструктура Discord вже є, ADR-023), але перетворює сирі дані на щотижневий список рішень.
- Правило процесу: **кожен пункт digest'а або стає issue, або явно позначається «accepted noise»**. Це і є механізм самовдосконалення — не магія, а конвеєр даних у Roadmap.
- LLM-складова: digest-markdown — ідеальний вхід для дешевої моделі («підсумуй, згрупуй, запропонуй пріоритет»), а раз на місяць — для сильної («чи не час міняти архітектуру X, бо дані показують Y»). Див. `docs/15_LLM_PROMPTS.md`.

### 4.5. GDPR-межі (узгоджено з ADR-006)

`events` і `export_feedback` — без email/IP; `session_hash` — добовий salt, непереслідуваний; retention 12 місяців; згадка у Privacy Policy. Параметри виробів — технічні дані, не PII.

## 5. Анти-цілі (що свідомо НЕ робимо)

На MS21 (2 vCPU / 4 GB) і 10 users/day **не потрібні**: microservices, Kubernetes, Kafka/event-sourcing, Prometheus+Grafana-стек, OpenTelemetry-колектор, ML-пайплайни. Sentry (SaaS free tier) + Postgres-таблиця events + Umami (self-hosted) + Discord webhook покривають 100% потреб MVP і v1.x. Складність — головний ворог solo-проєкту (R-07).
