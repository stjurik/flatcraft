[AI bugfix flow v2 — ЛОКАЛЬНЕ мультиагентне рев'ю draft PR зі Стадії "Фікс"
(ADR-035 §3). Запускає yurii ОДНІЄЮ командою на кожен ai-fix PR, у СВІЖІЙ
Claude Code-сесії:

Виконай docs/promts/ai-review-local.md для PR #<N>

(підставити реальний номер PR замість <N> — це єдиний параметр цього
промпту).]

РОЛЬ
Ти — Оркестратор рев'ю (docs/15 §0). Код фікса НЕ правиш. Писати можеш ЛИШЕ
тест-файли (`*.test.ts`, `*.test.tsx`, `*.spec.ts`, `workers/cad/tests/**`) —
і лише у гілку фікса (`fix/issue-{N}`), не в `main`. `agy` — Рев'юер і
Тест-інженер, ОБИДВІ ролі read-only (ADR-035 §4): пише лише у
`docs/promts/inputs/`, ніколи не редагує код фікса.

ЧИННІ ПРАВИЛА agy (без змін, docs/promts/agy-orchestration-recommendations.md)

- headless `agy -p`, ніколи інтерактивно;
- файлова передача — `agy -p` НЕ читає stdin, пиши контекст у файл і проси
  agy читати через `read_file`;
- явно забороняй bash/terminal-інструменти в кожному промпті agy;
- запис ЛИШЕ у `docs/promts/inputs/` — після КОЖНОГО виклику `git status`,
  стороння зміна деінде → `git restore` + запиши інцидент у фінальний
  коментар (не мовчки);
- verify-then-write: кожен вихід agy перевір проти реального коду ПЕРЕД
  використанням.

КРОКИ

1. **Diff → файл.**
   `gh pr diff <N> > docs/promts/inputs/_review-pr-<N>.diff` (файл — тимчасовий,
   не комітиться; видали його наприкінці). Прочитай сам PR-опис
   (`gh pr view <N> --json title,body,url`) і пов'язаний issue (номер із
   `Closes #M` у body) — це твій ПЛАН, проти якого звіряєш diff.

2. **agy-РЕВ'Ю (окремий виклик).**
   `agy` читає diff-файл (`read_file`) + сам план (встав текст issue/тріаж-
   коментаря прямо в промпт agy — коротко, це вже не untrusted, ти сам його
   прочитав і довіряєш) → вердикт:
   - відповідність diff'а плану з тріажу;
   - інваріанти CLAUDE.md §7 (CAD-обмеження) і §13 (must-not-break);
   - регресійний тест присутній І осмислений (не тавтологія «expect(true).toBe(true)»);
   - які ділянки коду зачеплені, чи є щось поза заявленим планом.

   Вердикт ✅/⚠️/❌ + список зауважень → `docs/promts/inputs/review-issue-<N>.md`.
   Постав явну заборону bash у промпті agy (правило вище).

   Опублікуй результат **ДОСЛІВНО** коментарем на PR з позначкою
   «**Gemini Reviewer (unedited)**» — без редагування тексту agy. Свою
   думку додай ОКРЕМИМ коментарем ЛИШЕ якщо бачиш ФАКТИЧНУ помилку у
   вердикті agy (розбіжність моделей — цінний сигнал сам собою, не привід
   мовчки виправляти чужий вердикт).

3. **agy-ТЕСТ-ІНЖЕНЕР (окремий виклик, після кроку 2).**
   За тим самим diff-файлом і релевантними Zod/Pydantic-схемами параметрів
   (передай agy шляхи, не вставляй увесь код у промпт) — запропонуй 5-10
   додаткових тест-кейсів для зачепленої зони: граничні значення (min/max
   з CLAUDE.md §7 таблиці), negative cases (щось за межею → очікувана
   помилка валідації), edge cases що діагональ diff'а міг пропустити.
   Кожен кейс — з очікуванням (input → expected). → `docs/promts/inputs/testcases-issue-<N>.md`.

   **ВЕРИФІКУЙ кожен кейс сам** проти реальних схем/валідаторів
   (verify-then-write — agy пропонує кейси текстом, не бачачи, чи вони
   взагалі мають сенс у контексті повної схеми):
   - Осмислені й коректні → оформи кодом, ОДНИМ окремим комітом у гілку
     фікса: `test(issue-<N>): additional cases (Gemini-proposed, Claude-verified)`.
   - Відхилені (дублюють наявний тест, не мають сенсу, невірне очікування)
     → перелічи з причиною відхилення у фінальному коментарі (крок 5), не
     мовчки викинь.
   - **Кейс, що ПАДАЄ на поточному коді фікса** = знахідка реального бага,
     не «тест написаний неправильно». НЕ підганяй очікування під поведінку,
     щоб тест позеленів. Замість цього: залиш тест червоним у коміті,
     напиши коментар на PR з описом знахідки, і **СТОП** — далі не
     продовжуй рев'ю, це блокер для yurii.

4. **Гігієна після кожного виклику agy.**
   `git status --short` — якщо змінилось щось ПОЗА
   `docs/promts/inputs/*`/твоїм тестовим комітом — це інцидент (не перший:
   Master Run 8 мав 2, задокументовані у `docs/promts/agy-orchestration-
recommendations.md` §4). `git restore` стороннє, ПОТІМ продовжуй.
   Допиши рядок у `docs/promts/inputs/agy-stats.md`:
   `дата | issue-<N> (review\|testcases) | модель agy | ok/scope-creep/timeout/auth | вердикт підтверджений (так/ні/n/a) | кейсів прийнято/відхилено`.

5. **Фінальний коментар на PR.**
   Підсумуй: вердикт agy (✅/⚠️/❌, коротко), скільки кейсів прийнято/
   відхилено (з причинами відхилення), твоя незгода з agy — ОКРЕМИМ
   абзацом, якщо була. Приклей посилання на `docs/promts/inputs/agy-stats.md`
   рядок цього прогону. Заверши явно: «Merge — рішення yurii після
   прочитання цього рев'ю.» НЕ мерджи сам, НЕ став жодних лейблів, які
   означали б approve.

Прибери тимчасовий `docs/promts/inputs/_review-pr-<N>.diff` наприкінці
(`git status` має лишитись чистим, крім твого одного тестового коміту в
гілці фікса — якщо він був).

STOP після фінального коментаря. Це і є кінець рев'ю-циклу; наступний крок —
merge, і це виключно yurii.
