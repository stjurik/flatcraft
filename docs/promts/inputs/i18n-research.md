### Статус next-intl у Next.js 15 (2026)

Станом на 2026 рік, `next-intl` (поточні стабільні версії 3.x) повністю підтримує Next.js 15 App Router як у stable, так і в canary гілках, включаючи нативну інтеграцію з React Server Components та Server Actions. Основною вимогою для Next.js 15 є необхідність використання асинхронних `params` (наприклад, `await params`), що успішно підтримується бібліотекою. Відомі проблеми з маршрутизацією найчастіше виникають при використанні стратегії `localePrefix: 'as-needed'`, де можуть траплятися непередбачувані скидання локалі через конфлікти між клієнтським станом та серверним middleware (особливо з жорстко заданими шляхами). Також існують певні нюанси сумісності з Turbopack, які зазвичай швидко вирішуються конфігурацією об'єкта `turbopack` у `next.config.ts`.

Порівняно зі стратегією «власних словників» (dictionary pattern без npm-залежностей), `next-intl` пропонує значно потужніший функціонал з коробки: готове middleware, плюралізацію, підтримку ICU-синтаксису та вбудоване форматування. З іншого боку, власні словники (`dictionaries.ts` + сегмент `[locale]`) забезпечують повну відсутність сторонніх залежностей, максимальну прозорість коду та дещо менший розмір бандлу, але вимагають ручної реалізації логіки визначення локалей. Таким чином, власні словники є кращим варіантом для простих проєктів із базовим перекладом, тоді як `next-intl` краще підходить для масштабованих додатків із комплексними вимогами до інтернаціоналізації.
