# 04. Risks Register

> Risk = подія, що може зруйнувати продукт або спалити ресурс. Кожен ризик: ймовірність × вплив = пріоритет. Mitigation = конкретний план.

Шкала: ймовірність 1–5, вплив 1–5, пріоритет = добуток.

---

## R-01. Неправильний K-фактор → бракована партія у виробника

| Поле        | Значення  |
| ----------- | --------- |
| Ймовірність | 4         |
| Вплив       | 5         |
| Пріоритет   | 20 (HIGH) |

**Опис:** K-фактор у `bend-machine-esi.yaml` — емпіричний дефолт за DIN 6935. Реальна машина ESI може давати ±0.05 відхилення → розгортка на 0.5–2 мм коротша/довша → готова деталь не сходиться по розмірах.

**Mitigation:**

1. Watermark на DXF: «Розгортка з емпіричного K-фактора, перевіряйте перед різкою. У випадку розбіжностей — повідомте на feedback@…»
2. Phase 5 пілот з ESI: 5 тестових деталей різної геометрії → заміри → калібрування таблиці.
3. У PDF-креслі — окрема нота для виробника: «K=0.40, deviation tolerance ±2мм per bend».
4. v1.1: API «report deviation» → користувач/виробник кидає реальні розміри → ми оновлюємо таблицю.

**Owner:** замовник (потрібен інженер з виробництва).

## R-02. 3D-preview і студія на слабких / мобільних пристроях

| Поле        | Значення   |
| ----------- | ---------- |
| Ймовірність | 3          |
| Вплив       | 3          |
| Пріоритет   | 9 (MEDIUM) |

**Опис:** Цільова аудиторія — DIY-люди у майстерні/гаражі, які часто заходять з телефону (рішення Phase 2.11, ADR-016). На mobile/слабких ноутбуках R3F-сцена (Phase 2.6, three.js Shape + ExtrudeGeometry — див. ADR-013) може гальмувати: великі parametric updates, дорогі shadows/HDR, недостатній touch-affordance у формах → користувач закриває вкладку.

ADR-013 уже зняв ризик 30-МБ WASM bundle (OpenCascade.js відкладено), тож ризик тепер про **загальну якість UX** mobile-студії, а не про конкретну бібліотеку.

**Mitigation:** **Progressive enhancement** — мобільний клієнт отримує спрощену версію, десктоп — повну. **Впроваджено у Phase 2.14.a** через pure `viewportQuality({isMobile, reduced})` helper у `@flatcraft/ui` (5 unit) + `useIsMobile`/`useReducedMotion` хуки. Адаптивна матриця:

|                                             | desktop  | mobile (≤767px)        | reduced-motion |
| ------------------------------------------- | -------- | ---------------------- | -------------- |
| Canvas `dpr`                                | `[1, 2]` | `[1, 1.5]`             | `[1, 1]`       |
| OrbitControls `enableZoom`                  | true     | false (pinch conflict) | false          |
| OrbitControls `enableRotate`                | true     | true                   | false          |
| `useDebouncedValue` ms                      | 100      | 250                    | 400            |
| `ExtrudeGeometry.curveSegments` (L-bracket) | 12       | 8                      | 6              |

Застосовано до всіх 5 scenes + 5 studios. Тапи ≥ 44×44 — інваріант з Phase 2.11/2.12.a (перевіряється у Playwright). Mobile-first layout (editor→viewport стек на xs/sm/md, 2-кол на lg+) + anchor «↓ Подивитися 3D-прев'ю» (`apps/web/src/components/studio-preview-anchor.tsx`, `lg:hidden`) для швидкого скролу до canvas.

**Round-bend cross-section** на усіх scenes з гибами реалізовано у **Phase 2.14.b**:

- `buildLBracketShapeCommands` (Phase 2.6, ExtrudeGeometry+Shape з 1 absarc) — застосовується у L-bracket і reuse у corner_angle.
- `buildZBracketShapeCommands` (Phase 2.14.b) — Shape з 2 absarc (bottom→middle, middle→top).
- `buildWallShelfShapeCommands` (Phase 2.14.b) — Shape з 1 або 2 absarc (back→shelf завжди; shelf→lip умовно при front_lip>0).

`perforated_panel` — плоский лист без гибів. Усі 5 scenes тепер показують реалістичні round inner bends у перерізі.

**Performance budget** — TTI < 3 c, parametric update < 200мс (CLAUDE.md §9). Якщо реальна аналітика покаже регресію на mobile — переоцінюємо матрицю.

**Acceptance:** на iPhone SE / Galaxy A-series студія L-bracket має рендеритись + давати export ≤ 6 сек end-to-end (валідація — реальний user feedback, не CI).

## R-03. Анти-зловживання лімітом «10 безкоштовних»

| Поле        | Значення    |
| ----------- | ----------- |
| Ймовірність | 5           |
| Вплив       | 2           |
| Пріоритет   | 10 (MEDIUM) |

**Опис:** Користувач створить 5 акаунтів з різних email-адрес, обійде ліміт. Замовник позиціонує платформу як соціальну, тож не хочемо drakon-захисту, але масова зловживання може покрити ресурс.

**Mitigation:**

1. Лічильник на email (verified) + IP — м'який сигнал. Без email-verify не дамо export.
2. Soft-throttling: якщо з одного IP за добу >5 нових email — зменшуємо ліміт до 3.
3. Honest-by-design: явно показуємо «це соціальна платформа, ваш донат йде на ЗСУ. Просимо чесно».
4. **Замовник свідомо приймає цей ризик** (відповідь у опитувальнику: «лишаємо на совість користувача»).

## R-04. GDPR штраф через витік PII

| Поле        | Значення    |
| ----------- | ----------- |
| Ймовірність | 2           |
| Вплив       | 5           |
| Пріоритет   | 10 (MEDIUM) |

**Опис:** Витік emails / IP / паролів. Навіть як хобі-проєкт у ЄС → штраф до €20k.

**Mitigation:**

1. Argon2id для паролів (а не bcrypt/sha).
2. Pino redact + Sentry beforeSend → PII у логи не потрапляє.
3. Postgres — слухає тільки на localhost/Docker-мережі сервера, назовні не відкритий (тільки api/worker).
4. Backups зашифровані age (`age` CLI) перед заливкою у R2.
5. Pen-test через `npm audit` + `pip-audit` перед кожним релізом, autoremediation у CI.
6. Залежності оновлюємо через Renovate (бот, щотижневі PR).
7. **Data residency:** після ADR-011 дані фізично в Україні (ДЦ Mirohost у Києві). Україна — «третя країна» для GDPR. Mitigation: у Privacy Policy явно вказуємо розташування даних + застосовуємо Standard Contractual Clauses; беремо явну згоду користувача при реєстрації. Бекапи в R2 (поза Україною) — це окреме сховище, теж згадуємо у Privacy Policy.

## R-05. Юрисдикція донатів (ЄС-користувачі)

| Поле        | Значення         |
| ----------- | ---------------- |
| Ймовірність | 3                |
| Вплив       | 4                |
| Пріоритет   | 12 (MEDIUM-HIGH) |

**Опис:** Якщо ЄС-користувач донатить «за послугу» → це підлягає VAT. Потрібно чітко: «це добровільний внесок на ЗСУ через посилання, платформа не виступає одержувачем коштів».

**Mitigation:**

1. ToS: явно «платформа не приймає платежі за послуги; усі донати йдуть напряму до фондів (Monobank Banka / UNITED24); продовження ліміту — це знак подяки нашого боку, а не зустрічна послуга».
2. UNITED24 для ЄС, бо вони офіційний urząd Зеленського і легко легально позиціонується.
3. Юрист-консультація перед запуском у ЄС (закладемо в Phase 5).
4. Якщо платформа взлетить — реєструємо як non-profit (UA: ГО, EU: stichting/Verein).

| ----------- | --------- |

## R-08. Залежність від OpenCascade.js / CadQuery (single point of dep)

| Поле        | Значення   |
| ----------- | ---------- |
| Ймовірність | 2          |
| Вплив       | 4          |
| Пріоритет   | 8 (MEDIUM) |

**Опис:** Якщо проєкт OpenCascade.js / CadQuery застаріє або зламається → треба портувати на Manifold або власний solver. Може забрати 1–2 місяці.

**Mitigation:**

1. Абстракція `cad-engine` — інтерфейси `SheetMetalSolver`, `Exporter`, `Validator`. Реалізації змінюються незалежно.
2. Тести через інтерфейси → перехід на іншу бібліотеку = переписати реалізацію, тести самі підкажуть, де поламано.
3. Обидві бібліотеки активно розвиваються станом на 2026 → ризик низький.

## R-11. Single-server staging/prod — збій хоста/диска/OOM = повний простій

| Поле        | Значення         |
| ----------- | ---------------- |
| Ймовірність | 3                |
| Вплив       | 4                |
| Пріоритет   | 12 (MEDIUM-HIGH) |

**Опис:** Весь стек (postgres/redis/cad-worker/api/web/caddy) живе на одному Mirohost MS21 (2 vCPU / 4 GB / 40 GB, ADR-011). HA немає: збій хоста, заповнення диска, OOM cad-worker'а (CadQuery важкий) або втрата SSH-ключа/vault-пароля → повний простій або заблокований deploy. Бекап раз/добу (pg_dump → R2), тож RPO до 24 год.

**Mitigation:**

1. Monitoring (Phase 5C): disk >80% + контейнер-health → Discord webhook (до OOM/full-disk).
2. `mem_limits` у compose (cad-worker 1.5g) — важка CAD-операція не вбиває api/postgres.
3. Backups cron 03:00 → age-encrypt → R2 (recovery — `docs/08_DEPLOYMENT.md` §5.5), локальний retention 3 дні.
4. Deploy відтворюваний з git+vault (Ansible) — host перестворюється за runbook'ом ~30 хв.
5. Vault-пароль + age-приватний ключ зберігаються **поза сервером** (інакше при втраті хоста втрачаємо і бекапи).
6. Production (Roadmap 5.10): розглянути managed Postgres або окремий DB-вузол, якщо навантаження зросте.

## R-12. Validator-bypass у export-pipeline → експорт неможливих деталей

| Поле        | Значення    |
| ----------- | ----------- |
| Ймовірність | 2           |
| Вплив       | 5           |
| Пріоритет   | 10 (MEDIUM) |

**Опис:** Виявлено (Hotfix 2.10.e): Z-bracket з `thickness=5.0` + `bend_radius=2.5` пройшов експорт і PDF потрапив у R2, хоча матриця `bend-machine-esi.yaml` для t=5 дозволяє лише R∈{4.0, 5.0}. Порушено інваріант CLAUDE.md §7 п.2 («Радіус гиба ≥ мін. для (матеріал, товщина)»). Root cause: `cad-engine validateBend` (коректний) виконувався лише браузер-side як UX-підсвічення; **серверний шлях експорту його не викликав**. Fastify `POST /exports` валідував лише Zod-shape і форвардив у cad-worker; Python-воркер перевіряв радіус лише проти **глобального** набору {1,2.5,4,5}, не проти thickness-залежної матриці. Наслідок: виробник може отримати неможливу деталь (брак, ризик безпеки платформи — соціальний проєкт на ЗСУ).

**Mitigation:**

1. Серверний gate у `apps/api` (ADR-019): `validateExportBends` через cad-engine `validateBend` ПЕРЕД постановкою job/forward → 422 RFC 9457 з `code: RADIUS_NOT_ALLOWED`, жодного артефакта у R2.
2. Python parity-валідатор (`flatcraft_cad/validate`) читає той самий YAML, відмовляє на старті `/export`-handler ДО CAD-операції — остання лінія оборони (прямий виклик воркера, майбутній BullMQ-producer).
3. Property-based тести (fast-check TS + hypothesis Python, по 1000 ітерацій) проти спільного oracle на одному YAML — закривають весь клас «радіус допустимий глобально, але не для цієї товщини».
4. Integration-тест API → 422 без створення артефактів.
5. **Не зроблено (поза hotfix):** аудит R2 на історичні invalid-експорти до фіксу — окремий runbook (Roadmap).
