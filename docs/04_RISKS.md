# 04. Risks Register

> Risk = подія, що може зруйнувати продукт або спалити ресурс. Кожен ризик: ймовірність × вплив = пріоритет. Mitigation = конкретний план.

Шкала: ймовірність 1–5, вплив 1–5, пріоритет = добуток.

---

## R-01. Неправильний K-фактор → бракована партія у виробника

| Поле | Значення |
|---|---|
| Ймовірність | 4 |
| Вплив | 5 |
| Пріоритет | 20 (HIGH) |

**Опис:** K-фактор у `bend-machine-esi.yaml` — емпіричний дефолт за DIN 6935. Реальна машина ESI може давати ±0.05 відхилення → розгортка на 0.5–2 мм коротша/довша → готова деталь не сходиться по розмірах.

**Mitigation:**
1. Watermark на DXF: «Розгортка з емпіричного K-фактора, перевіряйте перед різкою. У випадку розбіжностей — повідомте на feedback@…»
2. Phase 5 пілот з ESI: 5 тестових деталей різної геометрії → заміри → калібрування таблиці.
3. У PDF-креслі — окрема нота для виробника: «K=0.40, deviation tolerance ±2мм per bend».
4. v1.1: API «report deviation» → користувач/виробник кидає реальні розміри → ми оновлюємо таблицю.

**Owner:** замовник (потрібен інженер з виробництва).

## R-02. OpenCascade.js падає на слабких пристроях

| Поле | Значення |
|---|---|
| Ймовірність | 3 |
| Вплив | 3 |
| Пріоритет | 9 (MEDIUM) |

**Опис:** OpenCascade.js — wasm-bundle ~30 MB, parametric update може займати 500мс+ на старих ноутбуках/мобільних. Користувач закриє вкладку.

**Mitigation:**
1. Фічу `live update` обмежимо 30 fps debounce + Web Worker.
2. Fallback на server-side preview: якщо клієнт повідомляє про повільність, переключаємося на серверний рендер mesh.
3. Mobile detection: на маленьких екранах показуємо «desktop required for editing» (зменшуємо очікування).
4. Performance budget — TTI < 3 c (документовано у `CLAUDE.md`).

## R-03. Анти-зловживання лімітом «10 безкоштовних»

| Поле | Значення |
|---|---|
| Ймовірність | 5 |
| Вплив | 2 |
| Пріоритет | 10 (MEDIUM) |

**Опис:** Користувач створить 5 акаунтів з різних email-адрес, обійде ліміт. Замовник позиціонує платформу як соціальну, тож не хочемо drakon-захисту, але масова зловживання може покрити ресурс.

**Mitigation:**
1. Лічильник на email (verified) + IP — м'який сигнал. Без email-verify не дамо export.
2. Soft-throttling: якщо з одного IP за добу >5 нових email — зменшуємо ліміт до 3.
3. Honest-by-design: явно показуємо «це соціальна платформа, ваш донат йде на ЗСУ. Просимо чесно».
4. **Замовник свідомо приймає цей ризик** (відповідь у опитувальнику: «лишаємо на совість користувача»).

## R-04. GDPR штраф через витік PII

| Поле | Значення |
|---|---|
| Ймовірність | 2 |
| Вплив | 5 |
| Пріоритет | 10 (MEDIUM) |

**Опис:** Витік emails / IP / паролів. Навіть як хобі-проєкт у ЄС → штраф до €20k.

**Mitigation:**
1. Argon2id для паролів (а не bcrypt/sha).
2. Pino redact + Sentry beforeSend → PII у логи не потрапляє.
3. Postgres — слухає тільки на localhost/Docker-мережі сервера, назовні не відкритий (тільки api/worker).
4. Backups зашифровані age (`age` CLI) перед заливкою у R2.
5. Pen-test через `npm audit` + `pip-audit` перед кожним релізом, autoremediation у CI.
6. Залежності оновлюємо через Renovate (бот, щотижневі PR).
7. **Data residency:** після ADR-011 дані фізично в Україні (ДЦ Mirohost у Києві). Україна — «третя країна» для GDPR. Mitigation: у Privacy Policy явно вказуємо розташування даних + застосовуємо Standard Contractual Clauses; беремо явну згоду користувача при реєстрації. Бекапи в R2 (поза Україною) — це окреме сховище, теж згадуємо у Privacy Policy.

## R-05. Юрисдикція донатів (ЄС-користувачі)

| Поле | Значення |
|---|---|
| Ймовірність | 3 |
| Вплив | 4 |
| Пріоритет | 12 (MEDIUM-HIGH) |

**Опис:** Якщо ЄС-користувач донатить «за послугу» → це підлягає VAT. Потрібно чітко: «це добровільний внесок на ЗСУ через посилання, платформа не виступає одержувачем коштів».

**Mitigation:**
1. ToS: явно «платформа не приймає платежі за послуги; усі донати йдуть напряму до фондів (Monobank Banka / UNITED24); продовження ліміту — це знак подяки нашого боку, а не зустрічна послуга».
2. UNITED24 для ЄС, бо вони офіційний urząd Зеленського і легко легально позиціонується.
3. Юрист-консультація перед запуском у ЄС (закладемо в Phase 5).
4. Якщо платформа взлетить — реєструємо як non-profit (UA: ГО, EU: stichting/Verein).

## R-07. Solo-розробник вигоряє / зайнятий → проєкт стане

| Поле | Значення |
|---|---|
| Ймовірність | 4 |
| Вплив | 4 |
| Пріоритет | 16 (HIGH) |

**Опис:** Один junior + Claude Code; реалістично 8–12 тижнів до MVP. Війна, життя, втома → проєкт може зависнути.

**Mitigation:**
1. Open Source MIT з першого коміту → community може підхопити.
2. Документація-першою — навіть якщо проєкт зупиниться, наступний дев підніме його за день.
3. Маленькі спринти 1–2 тижні (Roadmap) → завжди є finished increment.
4. Перший production deploy після Phase 1 (CAD core у CLI-режимі) — навіть як демо для community/інвестора.

## R-08. Залежність від OpenCascade.js / CadQuery (single point of dep)

| Поле | Значення |
|---|---|
| Ймовірність | 2 |
| Вплив | 4 |
| Пріоритет | 8 (MEDIUM) |

**Опис:** Якщо проєкт OpenCascade.js / CadQuery застаріє або зламається → треба портувати на Manifold або власний solver. Може забрати 1–2 місяці.

**Mitigation:**
1. Абстракція `cad-engine` — інтерфейси `SheetMetalSolver`, `Exporter`, `Validator`. Реалізації змінюються незалежно.
2. Тести через інтерфейси → перехід на іншу бібліотеку = переписати реалізацію, тести самі підкажуть, де поламано.
3. Обидві бібліотеки активно розвиваються станом на 2026 → ризик низький.

## R-09. Юзер скачав DXF, виробник зробив брак — шукають винного у нас

| Поле | Значення |
|---|---|
| Ймовірність | 3 |
| Вплив | 3 |
| Пріоритет | 9 (MEDIUM) |

**Опис:** Розгортка зайшла, але виробник недотримався радіусу або кута → провина непрозора. Користувач напише негативний відгук.

**Mitigation:**
1. У PDF — окремий блок «Перевірити перед різкою»: контрольні розміри, K-фактор, мін радіус, мін полиця.
2. ToS: «Платформа надає рекомендаційні креслення; перевірка перед запуском у виробництво — на користувачі/виробнику».
3. Discord-канал `#feedback` для розборок live-кейсів.
4. v2: бібліотека верифікованих виробників, які підтвердили сумісність з нашими DXF.

## R-10. Вибір моделі (template parameters) не задовольняє реальні потреби

| Поле | Значення |
|---|---|
| Ймовірність | 4 |
| Вплив | 3 |
| Пріоритет | 12 (MEDIUM-HIGH) |

**Опис:** Користувачі захочуть кастомізації, що виходить за межі шаблонів (напр., нестандартні отвори, відгини). MVP не тримає це, юзер піде до конкурента.

**Mitigation:**
1. Phase 2: 5 шаблонів _з широкою параметризацією_ (отвори: кількість, розмір, патерн; полиці: довжина, кут, радіус; матеріал/товщина).
2. v1.1: drag-handles на 3D (level Б редагування).
3. Discord-форма «опиши, що тобі потрібно» → раз/місяць ми додаємо новий шаблон з топ-запитів.
